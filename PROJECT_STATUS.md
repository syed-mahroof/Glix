### Forgot Password via email OTP (2026-07-18, user-requested)
User asked what a forgot-password feature would need (OTP generation, mail sending, app passwords), then confirmed Gmail SMTP + an App Password over a transactional provider (Resend/SendGrid) after WebFetch-confirmed research showed those all require domain verification the user doesn't have (no domain owned). User then supplied a real Gmail address + App Password directly.

- **New `core/password_reset.py`** — no new model/migration; OTP codes and one-time reset tokens both live in Django's cache framework, already Redis-backed (`settings.CACHES`, added during Phase 25 for TMDB caching/throttling) rather than a new SQL table, since everything here is deliberately short-lived (10-minute TTL). `request_otp()` generates a 6-digit code, hashes it (SHA-256) before caching, and emails it via `send_mail`; `verify_otp()` checks the hash and mints a single-use `reset_token` (a second, shorter-lived cache entry) on success; `confirm_reset()` consumes that token and sets the new password.
- **Enumeration-safe by design:** `POST /auth/password-reset/request/` always returns the same generic 200 response whether or not the email is registered, and always sets the 60-second resend cooldown either way — the response and its timing can't be used to test which emails have accounts.
- **Brute-force protections:** a 60-second resend cooldown per email, a 5-attempt cap on wrong codes before the OTP is invalidated outright, and every code is single-use (deleted from cache the moment it's verified).
- **New endpoints**, added to `core/auth_views.py`/`core/urls.py`: `POST /auth/password-reset/request/` (email in, generic message out), `POST /auth/password-reset/verify/` (email + code in, `reset_token` out), `POST /auth/password-reset/confirm/` (`reset_token` + `new_password` in, the same `{access, refresh, profile}` envelope `RegisterView`/`LoginView` return — the user is logged straight back in after a successful reset, no separate re-login step).
- **Gmail SMTP wired via `EMAIL_*` settings** (`config/settings/base.py`, real credentials in `backend/.env`, placeholders in `.env.prod`) — `DEFAULT_FROM_EMAIL` deliberately defaults to the authenticated Gmail address itself (`Glix <EMAIL_HOST_USER>`), since Gmail rejects/rewrites a From header that isn't the account that authenticated, and Glix doesn't own a domain to send from a vanity address. Documented as a ~500/day-cap tradeoff, fine for testing/early users; swapping to a domain-backed transactional provider later only touches these two settings, not the `send_mail` call sites.
- **12 new tests** (`core/tests/test_password_reset.py`) — module-level (`request_otp`/`verify_otp`/`confirm_reset` directly) and full API round-trip via `APIClient`, covering: email sent only for a real registered account, silent no-op for an unknown one, resend cooldown throttling, wrong-code attempt counting, lockout after max attempts, single-use consumption of both the OTP and the reset token, and weak-password rejection on confirm. Real Redis cache used (matches production), but `EMAIL_BACKEND` overridden to Django's `locmem` backend per-test so tests never hit real Gmail SMTP — `django.core.mail.outbox` captures the send instead. Full suite (33 tests, prior 21 + these 12) passing in the live `watchtracker_backend` container with zero regressions.
- **Live-verified past the test suite**, not just `manage.py check`: rebuilt containers with `docker compose up -d --force-recreate` (the docker-restart-vs-recreate `.env`-reload gotcha from Phase 27 applies here too), then hit the real `/auth/password-reset/request/` endpoint against the user's actual email through the real Gmail SMTP relay — no error in the backend logs, confirming the App Password authenticates correctly against Gmail, not just against a mocked backend.
- **Frontend:** new `app/forgot-password.tsx`, a single screen with 3 internal steps (email → 6-digit code → new password) rather than 3 separate routes, since passing a `reset_token` between file-based routes via params is more fragile than local component state for a flow this short. Includes a resend-code button with a client-side 60-second cooldown countdown matching the backend's. A "Forgot password?" link was added to `login.tsx` above the sign-in button. On success, the screen stores the returned tokens and routes to `/loading` exactly like `login.tsx`'s own submit handler. `tsc --noEmit` confirmed at the same pre-existing error baseline, zero new errors from either new or touched files.

### EAS Android dev-client build failure — two Gradle duplicates + one runtime SDK mismatch (2026-07-18 → all root-caused 2026-07-19, found while testing Phase 27)
User hit `Invariant Violation: TurboModuleRegistry.getEnforcing(...): 'RNGoogleSignin' could not be found` at runtime — the installed dev-client APK predated `@react-native-google-signin/google-signin` being added to the project, so its native binary genuinely lacked the module (this is expected: adding any native module requires a new dev-client build, not just a JS reload). `expo-dev-client` itself also turned out to be missing from `package.json` entirely (added via `npx expo install expo-dev-client`) — without it, `eas build --profile development` fails outright before even reaching Gradle. So the crash is a symptom: fixing the blocked Android build IS the fix for the runtime error.

- Triggering the rebuild then surfaced the real blocker: Gradle's `:app:mergeDebugResources` failed with `Duplicate value for resource 'attr/actionBarSize'`. **The first diagnosis (a Material Components version conflict) was wrong** and three builds forcing `com.google.android.material`/`androidx.appcompat` versions had zero effect — proven by a local `:app:dependencies` run showing the graph already resolves to a single unified appcompat 1.7.0 / material 1.12.0, no version conflict anywhere for any pin to touch.
- **True root cause (confirmed by a local `:app:dependencies` + `:app:mergeDebugResources` run, not the masked EAS log):** `react-native-shared-preferences@1.0.2` (the Android home-screen-widget data bridge, `require`d in `store/watchStore.ts` and `widgets/android/WidgetProvider.tsx`) declares a vestigial `implementation "com.android.support:appcompat-v7:23.0.1"` in its own `android/build.gradle` — pre-AndroidX boilerplate from the old RN native-module template. Its Java imports **zero** `android.support.*` classes (only framework `android.content.SharedPreferences`), so the dependency is dead weight, but its resources still merge: the 2015-era support library ships `<declare-styleable name="Theme">` with a **full** `attr/actionBarSize` definition, colliding with AndroidX appcompat-1.7.0's own full `AppCompatTheme` `actionBarSize` — two full definitions of one attr = AAPT2 hard duplicate, merge dies. Safe and complete because the app is fully AndroidX (`android.useAndroidX=true`) and nothing legitimately uses the pre-AndroidX support stack.
- **Second blocker, surfaced only after Fix 1 unblocked the merge:** EAS build `da8a089d`'s raw Gradle log showed `:app:mergeDebugResources` **passing** and the build then dying at `:app:checkDebugDuplicateClasses` with `Duplicate class androidx.work.OneTimeWorkRequestKt (and siblings) found in work-runtime:2.8.1 and work-runtime-ktx:2.7.1`. Cause: WorkManager **2.8.0 merged the `work-runtime-ktx` Kotlin-extension classes into the main `work-runtime` artifact** (standalone `-ktx` became an empty stub), but `react-native-android-widget` still drags in `work-runtime-ktx:2.7.1`, whose real `*Kt` classes now duplicate the merged copies in `work-runtime:2.8.1`.
- **Fix 2 (same plugin):** added `resolutionStrategy { force 'androidx.work:work-runtime:2.8.1'; force 'androidx.work:work-runtime-ktx:2.8.1' }` to the injected `configurations.all` block, forcing the `-ktx` artifact to its empty 2.8.1 stub — duplicate removed, WorkManager runtime not downgraded.

### Sign in with Google/Apple (2026-07-16, user-requested)
User asked whether `django-allauth` was the best approach for SSO, and whether it's a good fit for Glix. **Recommended against it**: allauth is architected to be the OAuth client itself — a server-side, session-backed, redirect/callback flow with its own templates and a `django.contrib.sites` dependency, the right shape for a server-rendered web app. Glix is a pure Expo/React Native mobile app on a JWT-only, session-free DRF backend (locked per `AI_RULES.md`) — for mobile, the *device* is the OAuth client (native SDK hands the app an ID token directly), and the backend's only job is verifying that token and minting its own session, the same pattern Firebase Auth/Auth0/Supabase use server-side for mobile clients. Implemented as direct ID-token verification instead, using PyJWT (already pinned) — only one genuinely new dependency (`cryptography`, PyJWT's own RS256 backend, verified missing from the venv before adding it).

### Phase 6 widget deep-dive audit + repair (2026-07-14)
Traced the full widget data pipeline end-to-end (app.json plugin config → `store/watchStore.ts`'s `syncWidgetData` → `widgets/android/*` / `widgets/ios/*`) against the actual installed package APIs rather than assuming the existing code was wired correctly. It wasn't — found and fixed 6 real bugs:

1. **iOS data bridge was a complete no-op.** `syncWidgetData` wrote `widgetData.json` to `FileSystem.documentDirectory` — the main app's *private* sandbox directory, invisible to a widget extension process. The App Group entitlement (`group.com.watchtracker`) existed but nothing used it. Fixed: `syncWidgetData`/new `clearWidgetData` now call `Widget.updateSnapshot()` on the actual `expo-widgets` `Widget` instances (guarded `Platform.OS === 'ios'` + try/catch import, same defensive pattern as the existing `SharedPreferences` guard) — this is the SDK's real IPC mechanism; no shared-file storage needed at all.
2. **iOS widget files called `createWidget()` with the wrong signature entirely.** `widgets/ios/WatchlistWidget.tsx`/`UpcomingWidget.tsx` did `createWidget(WatchlistWidget, { name, description, supportedFamilies })` — a React component + a config object. The installed `expo-widgets` API (verified directly from `node_modules/expo-widgets/src/Widgets.ts`, not assumed) is `createWidget(name: string, layout: (props, environment) => JSX.Element)`, returning a `Widget` instance with `.updateSnapshot()`/`.updateTimeline()`/`.reload()`. This would have thrown/failed to register at the native layer on any real build. Rewrote both files to the correct signature, exporting the `Widget` instance (named, not default) so `watchStore.ts` can import and push data into it.
3. **`app.json`'s `expo-widgets` plugin config was empty** — just the bare string `"expo-widgets"`, so the plugin's `widgets` array (which the config plugin uses to generate the actual iOS widget extension target/Info.plist entries) defaulted to `[]`. The native widget extension was never being told "Watchlist" and "Upcoming" exist. Fixed: converted to a full config object with `groupIdentifier: "group.com.watchtracker"` (matching the entitlement) and both widgets declared under `ios.supportedFamilies`.
4. **Dead duplicate `syncWidgetData` in `widgets/android/WidgetProvider.tsx`** — exported but never imported anywhere (confirmed via repo-wide grep); the real write path was always `watchStore.ts`'s own copy. Removed the dead export to stop it looking like a second, competing write path.
5. **`syncWidgetData`'s data derivation was subtly wrong**, not just structurally broken: "next episode" used `.find(ep => !ep.is_watched)` (first unwatched in array order — no chronological guarantee, could surface an unaired future episode as "NEXT UP," contradicting the app's own unaired-episode guard elsewhere). "Upcoming" was `toWatch.filter(show => show.air_date)` — a coincidental subset of the watch-next bucket, not real upcoming-episode data. Fixed by extracting the Shows Hub's own `pickNextEpisode()` out of `(tabs)/index.tsx` into shared `lib/upcoming.ts` (now exported, used by both the screen and the store) and reusing the existing `buildUpcomingItems()` for the upcoming feed — the widget now shows exactly what the in-app Shows Hub/Upcoming tab would show, not an approximation.
6. **Widget data was never cleared on logout** — `settings.tsx`'s `performLogout` deleted the JWT tokens but left the previous session's watchlist sitting in `SharedPreferences`/the iOS widget snapshot indefinitely, visible to whoever uses the device next. Added `clearWidgetData()` to the store, called from `performLogout`'s `finally` block.

**Also, UX/polish (not bugs, but asked for explicitly):** Android widgets now proactively call `requestWidgetUpdate()` right after writing fresh data (safe no-op pre-native-build — verified `AndroidWidget.ts`'s no-op module never throws) instead of waiting for Android's own `updatePeriodMillis` interval, so a checkmark tap reflects on the home screen widget immediately rather than up to ~30 minutes later. Android widgets also gained a poster thumbnail (`ImageWidget`, previously imported but unused in `WatchlistWidget.tsx` and missing entirely from `UpcomingWidget.tsx`) to match the iOS layout's existing image treatment.

**Left as a known, documented gap (not silently patched):** `app.json`'s `expo.ios`/`expo.android` have no `bundleIdentifier`/`package` set at all. `expo-widgets`' plugin defaults the widget target's bundle id to `<main bundle identifier>.ExpoWidgetsTarget` — with no main identifier, `expo prebuild`/EAS build will not produce a coherent bundle id for the widget extension. Assigning one is a product/app-identity decision (reverse-DNS naming), not a wiring bug — flagged in `AUDIT.md` rather than guessed at.

### Discover Filter & Sort bug fix — real TMDB integration (2026-07-14, user live-tested)
The Filter & Sort bottom sheet in Discover looked fully functional (sort pills, genre pills, reset button all responded to taps) but was completely disconnected from any data — reported by the user as "when we select anything there is no filter."

- **Root cause:** `discover.tsx` filtered feed sections client-side with `(section.items ?? []).filter(() => true)` — a literal no-op that kept every item regardless of the selected genre. `sortOrder` was stored in `discoverStore` but never read anywhere. Neither the backend nor `TMDBService` had any support for TMDB's genre/sort parameters — `DiscoverFeedView` only ever returned a fixed set of curated sections (trending/popular/airing-today or trending/popular/top-rated/coming-soon).
- **Backend, built properly rather than faked:** new `TMDBService.discover_tv()`/`discover_movies()` wrapping TMDB's actual `/discover/{tv,movie}` endpoints (`with_genres`, `sort_by`, plus a `vote_count.gte=100` floor when sorting by rating — TMDB's vote-average sort otherwise surfaces obscure 1-vote "10/10" titles). New `DiscoverFilterView` (`GET /api/discover/filter/`). "Trending" needed special handling: TMDB's `/trending` endpoint has no `with_genres` param at all, so `get_trending()` gained an additive `include_genre_ids` flag (existing callers unaffected) and the view filters trending results by `genre_ids` in Python instead of silently degrading "Trending + Genre" to a plain popularity sort.
- **Live-verified against real TMDB** (not just typechecked): Drama sorted by rating returned plausible high-vote shows, Horror sorted by popularity returned real horror movies, and trending filtered to Sci-Fi & Fantasy correctly narrowed 20 trending items to 10 genuinely sci-fi/fantasy titles (Rick and Morty, House of the Dragon).
- **Frontend:** `discoverStore.ts` gained `filteredResults`/`fetchFilteredResults()`/`isFilterActive()`/`resetFilters()`; selecting a genre or non-default sort now actually refetches (or clears back to the curated feed if returning to fully-default). `discover.tsx` renders filtered results in the same grid as universal search (same response shape, same card component) instead of the broken curated-sections stub. Added a small badge dot on the filter icon so an active filter is visible even after the sheet is closed — previously there was no way to tell a filter was applied once you dismissed the sheet.
- Verified: `tsc --noEmit` — only the same pre-existing repo-wide FlashList `estimatedItemSize` issue, now also on the new filtered-results grid (consistent with every other FlashList in the app, not a new bug).
