### Remove/undo "Add to Watchlist" (2026-07-23, fix prompt's Phase F: adding a show/movie by accident, or wanting to reverse it, currently cannot be undone)

Read `backend/core/urls.py`/`views.py`/`models.py`, `client-mobile/store/watchStore.ts`, `client-mobile/app/show/[id].tsx`, `client-mobile/app/movie/[id].tsx`, `client-mobile/lib/useCatchupCascade.ts`, and `client-mobile/components/Snackbar.tsx` live first.

- **Confirmed the actual gap before building anything:** `urls.py` had `watchlist/add/` and `movies/add/` but no remove/delete counterpart at all for either — not a broken frontend handler, a genuinely missing endpoint on both sides.
- **Real data-modeling decision, surfaced rather than guessed:** the fix prompt explicitly flagged that "does remove also delete watch history?" needed a decision, not an assumption, since it has irreversible stats/badge consequences either way. Asked the user directly. Answer: **full delete** — removing a title wipes its `WatchState`/`MovieWatchState` rows too, not just the `Watchlist`/`MovieWatchlist` join row. - **Backend — `ShowRemoveView`/`MovieRemoveView`:** `DELETE /api/watchlist/remove/` (`{show_id}`) and `DELETE /api/movies/watchlist/remove/` (`{movie_id}`), both `IsAuthenticated`, both wrapped in `transaction.atomic()` with `select_for_update()` on the profile row — same locking pattern `WatchStateToggleView`/`BulkWatchStateToggleView` already use. Each captures every `WatchState`/`MovieWatchState` row for that title (server-side, not whatever the client happened to have cached — same reasoning as `CatchupCheckView`'s eager-cache-before-answering design) before deleting it, decrements `total_time_watched` via the identical `F()`-expression + non-negative-floor pattern the toggle views use, and **deliberately leaves `earned_badges` untouched** — badges are additive-only everywhere else in this codebase (an un-watch via the toggle endpoints never revokes one either), so a remove shouldn't be the one place that does.

- **Deliberately the same Undo pattern as the Catch-Up cascade** (act immediately, offer a real reversal) rather than a deferred-commit/grace-window design — `Snackbar` was already built exactly for "bulk/hard-to-reverse action + real Undo" by that feature, and the fix prompt asked to reuse it rather than build a second toast component.

### TV Time import: chunked/resumable + diff-aware re-import (2026-07-23, fix prompt's Phase B: a ~300+ show library hits `SoftTimeLimitExceeded` and re-running blindly reprocesses everything)

Read the live import pipeline end-to-end first — `backend/core/tasks.py`'s `run_tvtime_import`/`_import_one_show`/`_import_one_movie`, `backend/core/views.py`'s `TVTimeImportView`, `backend/core/models.py`'s `ImportJob` — before touching anything.

- **Root cause (no clean resume):** the whole export (up to ~1,100 sequential TMDB calls for a full 200-series library) ran as one Celery invocation under a single `soft_time_limit=1500`/`time_limit=1560`. On a large-enough library the soft limit fired mid-run; whether that got caught by the per-item `except Exception` (wasting the rest of the run re-trying already-slow items) or the outer one (job marked FAILED), the code unconditionally did `job.payload = {}` at the very end regardless of status — destroying the only record of what still needed processing. - **Root cause (no diff-awareness):** `_import_one_show` already skipped DB-level duplicate writes (`already` = existing `WatchState` ids) but built `season_numbers` from the *full* `wanted` set before checking that, so a show already 100% imported and marked in a prior run still re-fetched every one of its watched seasons from TMDB (or at best the local cache) on every resubmission.
- **Fix — chunked + resumable:** `run_tvtime_import` now processes at most `IMPORT_CHUNK_SIZE=10` shows+movies per invocation (`job.processed` doubles as the resume cursor — it's saved to the DB after every single item, unchanged from before) and, if work remains, re-enqueues itself (`run_tvtime_import.apply_async(args=[job_id], countdown=1)`) instead of looping through the whole payload. `soft_time_limit`/`time_limit` dropped from 25min/26min to 5min/5.5min — sized for one chunk (~30s/item budget, ~20x the original average per-call pace), not the whole export. Only a job that finishes **every** chunk (`status=SUCCESS`) clears `payload`; a `FAILED` job (a chunk's soft-time-limit hit, or `TVTimeImportView`'s existing orphaned-job close-out) now keeps both `payload` and `processed` intact.
